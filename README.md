## Modul 1

### Refleksi 1
Berdasarkan hasil implementasi, beberapa prinsip clean code yang sudah diterapkan adalah meaningful names, dengan memberikan nama variabel yang deskriptif sehingga kode dapat lebih mudah dimengerti. Selain itu, setiap function diberi nama yang deskriptif dan hanya melakukan suatu pekerjaan yang sesuai dengan namanya. Kemudian, prinsip objects and data structures juga digunakan, interface digunakan dalam kode ini sehingga bagian lain dari aplikasi hanya perlu tahu fungsi yang tersedia tanpa harus mengetahui logika di dalamnya. Prinsip secure coding juga digunakan dalam implementasi, salah satunya adalah penggunaan UUID sebagai identifier produk untuk mencegah ID enumeration. Kemudian, diterapkan juga redirect-after-POST untuk memastikan tidak ada pengiriman data berulang yang tidak sengaja. Saya juga menemukan beberapa bagian yang dapat ditingkatkan, seperti validasi input saat menambahkan produk baru, nama produk bisa berupa string kosong dan kuantitas produk dapat berupa angka nol atau negatif.

### Refleksi 2
Setelah membuat unit test, saya merasa lebih yakin dengan kode yang telah dibuat karena setiap fungsi telah divalidasi secara otomatis. Jika suatu saat saya mengubah kode, maka saya bisa langsung mengetahui apabila ada fungsi yang rusak dengan menjalankan ulang unit test. Tidak ada jumlah unit test yang pasti dalam satu kelas, tetapi jumlahnya harus cukup untuk meng-cover seluruh kemungkinan logika dalam sebuah kelas. Setiap metode sebaiknya memiliki tes untuk positive case dan negative case. Untuk memastikan bahwa unit test yang dibuat sudah cukup untuk memastikan program berjalan dengan baik, maka kita dapat melihat code coverage. Jika code coverage sudah cukup baik, misalnya 80% atau bahkan 100%, maka test yang dilakukan sudah cukup luas. Akan tetapi, 100% code coverage belum tentu menjamin program yang dibuat bebas bug. Code coverage 100% hanya berarti semua baris kode pernah dijalankan oleh tes, namun belum tentu menjamin logika bisnisnya sudah benar untuk berbagai variasi input. Kesalahan logika atau bug tetap bisa terjadi meskipun seluruh baris sudah tercover.

Jika saya membuat functional test baru dengan prosedur setup dan variabel yang sama persis dengan CreateProductFunctionalTest.java, maka kualitas kode tersebut akan menurun karena beberapa masalah clean code. Akan terjadi pengulangan kode yang sama untuk konfigurasi @LocalServerPort, baseUrl, dan @BeforeEach di setiap kelas baru. Ini akan menyebabkan redundansi. Kemudian, jika suatu saat saya ingin mengubah base URL, maka saya harus mengubahnya satu per satu di file. Agar code menjadi lebih clean, sebaiknya membuat sebuah base class yang berisi semua konfigurasi umum dan prosedur setup. Sehingga, sebuah functional test baru cukup meng-extends dari base class tersebut dan kode menjadi lebih mudah dibaca dan mengikuti prinsip DRY (Don't Repeat Yourself).
